What Each Project Is
Read all 12 EigenView docs plus all EigenView materials from this conversation. Here is the honest summary of what each project has built and where each stops.
EigenView — This Conversation
Macro Regime + Multi-Signal Screener
A macro-first intelligence engine that runs a 4-factor regime gate (GEX, VIX term structure, DIX, breadth) before scoring any instrument. When regime is green, scores stocks AND futures across 5 signals. Output: a ranked curated list of 5–10 instruments with setup type, entry/stop, and conviction drivers.

Unique layers EigenView has:
• VIX term structure (backwardation / contango regime)
• DIX dark pool index (SqueezeMetrics macro signal)
• CTA flip level tracking (futures-specific)
• Commodity signal stack (OVX, COT, futures curve shape)
• Futures coverage — ES, NQ, CL, GC, SI natively
• Retail capitulation scoring (Citadel GMI logic)
• Macro regime gate that BLOCKS individual stock picks in RED regime

What EigenView lacks: No technical analysis layer. No dormant-bet radar. No LLM-written thesis per pick. No options structure suggestions. No conversation AI. No full app architecture. Exists as a proposal and a dashboard mockup — not code.
EigenView — Local Docs (12 files read)
Multi-Factor Options Pick Engine with Dormant Radar
A fully architected local application (Python/FastAPI/Vanilla JS) that fuses 5 factors into 3–5 curated options trade ideas per day. Has a complete build plan (10 phases), all engineering standards locked, design system defined, wireframes built, and decisions logged.

Unique layers EigenView has:
• Dormant-bet radar — tracks large long-dated positions opened 30–180 days ago, scores activation probability (THE differentiator, nobody else does this)
• Technical analysis factor — trend, momentum, volatility state, ML pattern classifier
• LLM-written thesis per pick ("why this, why now, what kills it")
• Novelty-weighted sentiment (embedding distance vs baseline — not raw news volume)
• Full options structure suggestion per pick (direction, expiry, R:R)
• AI chat dock — context-aware, knows selected pick
• Complete module framework — 21 UI modules, 5 preset templates
• Complete architecture: FastAPI + SQLite + scheduler + notifications

What EigenView lacks: No macro regime gate. No VIX term structure signal. No DIX signal. No CTA flip awareness. No commodity futures (US equities only, by design). No futures instruments at all. No retail capitulation layer. Individual picks can fire in a negative-GEX meltdown.
Why Merge
Each Fills the Other's Critical Gap
This isn't a "nice to have" merger. Each project has a hard structural weakness that the other directly solves. Without the merge, both products have a fatal flaw at launch.
EigenView's Fatal Flaw Without EigenView
Picks Fire in Meltdowns
EigenView's gate logic (TA + GEX + 2 of 3 remaining) operates at the individual stock level only. The macro regime is not checked before picks are output.

In September–April 2026, GEX was negative on individual names AND the macro regime was red. EigenView would have output picks. Those picks would have been long options buys. They would have been wrong — not because the stock logic was wrong, but because the macro regime made every long a headwind trade.

EigenView's regime gate fixes this completely. If SPX GEX + VIX term structure + DIX + breadth score RED, no long picks output. Full stop. This one addition prevents the Sep–Apr trap from recurring.
EigenView's Fatal Flaw Without EigenView
No Technical Analysis. No Thesis. No Structure.
EigenView scores instruments on options flow signals, GEX, VIX, and dark pool data. There is no technical analysis factor. A stock can score 9/10 on EigenView signals with a clean GEX flip cross and three dark pool prints — and be in a multi-month downtrend with every moving average stacked bearishly against it.

Additionally, EigenView outputs a trade card with entry/stop zones — but no thesis, no options structure suggestion, no "what kills this trade." A trader reading it knows what the signals are but not what to do with them in the options market specifically.

EigenView's TA factor and LLM thesis layer fix this completely. TA becomes a hard gate (just as it is in EigenView). The LLM thesis wraps the combined signal output in plain English with a specific options structure recommendation.

The Merged Product
Combined Signal Stack — 9 Layers
Every layer from both projects, deduplicated, assigned to the right scope. The gate logic becomes: Macro Gate GREEN → TA fires → Per-stock GEX fires → 2+ of remaining 5 align.
Gate 0 — Macro Regime (NEW from EigenView)
Runs Before Any Pick Is Scored
Inputs: SPX GEX regime (SpotGamma), VIX M1/M2 term structure (VIXCentral), DIX (SqueezeMetrics), SPX breadth (% above 50dma)

Score: 0–10. GREEN ≥7, YELLOW 5–6, RED ≤4.

Gate logic: RED → output short setups and cash signal only. No long picks regardless of individual stock signals. YELLOW → picks shown with "MACRO CAUTION" flag. GREEN → normal operation.

This is new to the merged product. EigenView had no macro gate. EigenView has it fully defined.
Gate 1 — Technical Analysis (from EigenView)
Hard Gate — No Pick Without TA Firing
Signals: 20/50/200 EMA stack, ADX, RSI + divergence, MACD, ATR, Bollinger bandwidth, compression detection, volume profile POC/HVN/LVN, swing high/low levels

Pattern classifier: compression_break | breakout | pullback_in_trend | flag | double_top | double_bottom | head_shoulders | vcp | exhaustion | range

Firing condition: pattern confidence >0.6 AND weekly trend doesn't contradict daily setup

Fully designed in EigenView docs 03. Unchanged in merge.
Gate 2 — Per-Stock GEX / Dealer Positioning (Merged)
Hard Gate — No Pick Without GEX Firing
Identical in both projects. EigenView and EigenView both score: net dealer gamma, gamma flip level, call wall, put wall, regime flag (short_gamma / long_gamma / flip_zone).

Merge decision: one implementation, used in both EigenView pick scoring AND EigenView futures scoring. Macro-level GEX (Gate 0) is separate from per-stock GEX (Gate 2).

Chart overlay: GEX levels rendered on price chart as in EigenView's price-chart module.
Factor 1 — Options Flow (Merged, Enhanced)
6-Signal Flow Factor (Both Projects Combined)
From EigenView: V/OI ratio (≥3 = new position), premium size (>$500K), aggressive side (ask vs bid), sweep detection

From EigenView: dark pool cluster detection (3+ prints in ±0.5% band at price level), intermarket sweep confirmation

Combined firing condition: ≥1 contract meets premium >$500K + V/OI ≥3 + aggressive side = pick direction. Dark pool cluster at entry zone = bonus conviction point.

This is a genuine improvement over either project alone.
Factor 2 — Dormant-Bet Radar (EigenView, Unchanged)
THE Moat
Large long-dated positions (30–180 days old, ≥$500K premium, ≥90 DTE at opening) scored for activation probability near catalyst. V1 rule-based. V2 ML classifier.

Nothing changes. This is EigenView's crown jewel and the hardest to replicate. EigenView gains it for free in the merge.
Factor 3 — Catalyst + Novelty Sentiment (EigenView, Unchanged)
Novelty-Weighted, Not Volume-Weighted
Embedding distance from 30-day rolling baseline per ticker. Catalyst proximity scoring (earnings, FDA, macro). LLM filter for MATERIAL vs NOISE. FinBERT direction scoring.

Nothing changes. EigenView's design is production-ready.
Factor 4 — Retail Capitulation + VIX Structure (EigenView, New)
Macro Timing Layer
VIX M1/M2 backwardation score + DIX trend direction + retail put premium extreme (Citadel GMI logic). These fire at the macro level and upgrade individual pick conviction when the macro environment is favorable for reversals.

New to EigenView. EigenView contributes this entire layer.

Product Identity
What to Call It
Two names on the table. One decision to make before Phase 0.
Option A — Keep "EigenView"
EigenView (Extended)
The EigenView brand is more descriptive of what the product does — multiple signals converging into one output. The repo is already named it. All 12 docs reference it. EigenView becomes a module name ("Macro Pulse" or "Regime Gate") inside the product.

Pros: zero renaming friction, docs stay valid, name matches the concept.
Cons: conflicts with Atlassian's EigenView wiki product — SEO problem if you ever go commercial.
Option B — Keep "EigenView"
EigenView (Extended)
EigenView is shorter, more brand-able, and has no naming conflicts. The macro regime layer IS the pulse of the market — it's a better metaphor for the macro-first approach.

Pros: stronger brand, no Atlassian conflict, works commercially.
Cons: requires renaming all 12 EigenView docs and the repo. One-time effort, ~2 hours.
Option C — New Name
Something New
If this becomes a commercial product, neither name may be optimal. Examples: SIGNAL (clean, generic), APEX (directional), REGIME (describes the macro-first approach), VECTOR (directional conviction).

Only worth considering if commercial launch is the plan from Day 1. For personal use: keep EigenView, add EigenView layers.
Recommendation
The Verdict
Merge. Use EigenView as the Foundation. Add EigenView on Top.
EigenView's architecture is production-ready. 12 docs, complete engineering standards, wireframe v2 locked, 10-phase build plan with acceptance criteria, all decisions logged. This is months of design work. Don't throw it away.

EigenView's macro regime gate is the missing layer. Without it, EigenView can output picks in a red-regime meltdown. That's the Sep–Apr trap. Adding the macro gate (4 free data sources, 1 new Python module, ~20 lines in synthesis) takes 6 hours and closes the most critical product risk.

EigenView's commodity + futures layer is optional but valuable. If you trade CL and ES in addition to equity options, add the futures module (~200 lines). If you trade equity options only, skip it for V1 — it's a clean add-on, not a dependency.

The total cost of the merge: +12 hours on top of EigenView's 15–20 hour original estimate. The result is a product with 9 signal layers, macro-first gating, dormant-bet radar, LLM thesis, AI chat, and commodity coverage — across stocks AND futures. Nothing on the market has all of this.

Start Phase 0 of EigenView as planned. Add the two new data fetchers in Phase 1. Add macro_regime.py alongside the existing factor modules in Phase 2a. Everything else follows the existing build plan unchanged.
