PR Lifecycle — Two-Tier System ★

TierBranch prefixWhen to usePhases requiredPR lifecycle
Feature feature/ New behavior, new factor, new UI module, any non-trivial change All 29 steps: research → spec → issue → stub → CI → design review → human gate → implement → merge Open as DRAFT at step 8. Ready-for-review at step 22. Human merges at step 28. One PR per feature, never reuse.
Fix fix/ Confirmed bug with known fix, typo, config update, dependency bump Lightweight: skip steps 5–14. Create branch → open PR → implement → tests pass → human merges. Pre-build-gate recognizes fix/ prefix and skips gate:awaiting-step21 check. Still requires open PR. Human still merges (no self-merge).

Novel Elements — Why They Were Critical

ElementStepsStandard practiceWhat was addedWhy critical
Pre-implementation spec lock5–8Spec in a doc, no enforcementSpec written to .boss/ BEFORE branch opens. Gate blocks src/ edits until PR exists.Without this, implementation starts before requirements are agreed. Review happens after code exists — too late to catch design problems.
pre-build-gate (GitHub API check)6No equivalentHook calls gh pr list before every file edit. Cannot be faked by Claude — GitHub API is the source of truth.Claude was editing master directly, bypassing all branch/PR controls. This closes the gap at the file-write level.
gate:awaiting-step21 label ★18, 21, 22No equivalentCI sets this label (GITHUB_TOKEN) after red phase verified. pre-build-gate reads label — blocks src/ edits until human removes it. Human removing = step 21 approval. Unfakeable.H9: Implementation could start before human approval. "Open PR" check wasn't enough — Claude could start coding immediately after opening the PR. The label creates a hard gate between stub commit and implementation.
stop-gate: dirty check first ★11, 24No equivalent★ Order fixed: dirty src/ check runs INSTANTLY before test suite. Block fires in <1s if uncommitted. Tests only run (90s) when src/ is clean.Prior order ran 90s test suite on every turn, even when the real issue was uncommitted src/ files. Now: dirty = instant block. Clean = run tests.
Pre-implementation CI audit13–17Post-implementation review onlyCI runs full audit of spec + ACs + stubs BEFORE implementation. Human approves before code is written.Finding a bad requirement after implementation = wasted work. Finding it at step 17 = zero code written yet.
Zero-context Claude-in-CI audit16, 26No equivalentSeparate Claude API call with no project memory audits the same artifacts the builder Claude wrote.If Claude writes tests AND verifies them in same session, it rationalizes its own bad tests. Zero-context = adversarial by construction.
Issue body AC format enforcement15ACs are guidance onlyCI fetches issue body via gh api. Fails build if GIVEN/THEN/AC1/AC2 not present.Weak ACs (one-liner descriptions) produce untestable requirements. This makes GIVEN/THEN format a hard gate, not a suggestion.
Two-tier PR system ★AllOne-size-fits-all PRsfeature/ = full 29 steps. fix/ = lightweight (skips spec/stub phases, gate:awaiting-step21 exempt). Detection by branch prefix.Bug fixes shouldn't require full spec/stub cycle. But they still need a branch, PR, and human merge — no direct master commits.
Phase-aware pre-commit hook10, 23All-or-nothing: tests must passHook detects phase from git staging area. RED: stubs must FAIL. GREEN: all tests must pass. Wrong phase = commit blocked.Standard pre-commit blocks red phase (stubs fail = block). Need to invert: stubs failing is correct in red phase. Phase detection from staging area is unfakeable by Claude.
Per-feature step tracker ★AllNo step tracking.boss/CURRENT-STEP.md updated after every step. GitHub labels as external source of truth. Claude fills template at step 5.Without tracking, Claude loses step state across turns. External labels (set by CI/human, not Claude) are the authoritative step state.
