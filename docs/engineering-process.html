Novel Elements — Why They Were Critical

ElementStepsStandard practiceWhat was addedWhy critical
Pre-implementation spec lock5–8Spec in a doc, no enforcementSpec written to .boss/ BEFORE branch opens. Gate blocks src/ edits until PR exists.Without this, implementation starts before requirements are agreed. Review happens after code exists — too late to catch design problems.
pre-build-gate (GitHub API check)6No equivalentHook calls gh pr list before every file edit. Cannot be faked by Claude — GitHub API is the source of truth.Claude was editing master directly, bypassing all branch/PR controls. This closes the gap at the file-write level.
stop-gate dirty-file check11, 24No equivalentEnd-of-turn hook checks git diff. Blocks turn if src/ has uncommitted changes.Claude could write files and never commit — pre-commit never fires, CI never runs, changes invisible. This forces commit before claiming done.
Pre-implementation CI audit15–19Post-implementation review onlyCI runs full audit of spec + ACs + stubs BEFORE implementation. Human approves before code is written.Finding a bad requirement after implementation = wasted work. Finding it at step 19 = zero code written yet.
Zero-context Claude-in-CI audit18, 26No equivalentSeparate Claude API call with no project memory audits the same artifacts the builder Claude wrote.If Claude writes tests AND verifies them in same session, it rationalizes its own bad tests. Zero-context = adversarial by construction.
Issue body AC format enforcement17ACs are guidance onlyCI fetches issue body via gh api. Fails build if GIVEN/THEN/AC1/AC2 not present.Weak ACs (one-liner descriptions) produce untestable requirements. This makes GIVEN/THEN format a hard gate, not a suggestion.
